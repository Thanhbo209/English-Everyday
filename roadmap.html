Tenant Isolation Strategy

ApproachUsed WhenIsolation Level
Row-Level (tenant_id)MVP → SaaSMedium
PG Schema per TenantGrowth → ScaleHigh
DB per TenantEnterprise OnlyFull

📌 Design Decision: Start with tenant_id row-level isolation. Every table carries tenant_id UUID NOT NULL. This costs nothing in Phase 1 but enables schema migration later.

📌 Why not schema-per-tenant from day one? Migration complexity and Prisma limitations. Row-level with RLS (Row-Level Security) gives 95% of the isolation guarantees at 5% of the cost.

📌 Future path: When an enterprise client requires full isolation, migrate their tenant_id rows into a dedicated schema using pg_dump --schema.

-- TENANTS — Root of multi-tenancy. Every other table
-- carries tenant_id to enable row-level scoping.
-- ═══════════════════════════════════════════════════════
CREATE TABLE tenants (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug        VARCHAR(63) UNIQUE NOT NULL, -- used in subdomain routing
  name        VARCHAR(255) NOT NULL,
  plan        VARCHAR(50) NOT NULL DEFAULT 'free', -- free|starter|pro|enterprise
  status      VARCHAR(20) NOT NULL DEFAULT 'active',
  settings    JSONB NOT NULL DEFAULT '{}', -- branding, feature flags, quotas
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_tenants_slug ON tenants(slug);
CREATE INDEX idx_tenants_plan ON tenants(plan);

-- ═══════════════════════════════════════════════════════
-- USERS — Unified user model across all roles.
-- Role is NOT a column — it's a relationship via user_roles.
-- This allows a user to be Teacher in one classroom,
-- Student in another (cross-enrollment).
-- ═══════════════════════════════════════════════════════
CREATE TABLE users (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id      UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  org_id         UUID REFERENCES organizations(id),
  email          VARCHAR(320) NOT NULL,
  password_hash  VARCHAR(72) NOT NULL, -- bcrypt cost 12
  first_name     VARCHAR(100) NOT NULL,
  last_name      VARCHAR(100) NOT NULL,
  avatar_url     TEXT,
  status         VARCHAR(20) NOT NULL DEFAULT 'active',
  preferences    JSONB NOT NULL DEFAULT '{}',
  email_verified BOOLEAN NOT NULL DEFAULT false,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(tenant_id, email) -- email unique per tenant, not globally
);
CREATE INDEX idx_users_tenant ON users(tenant_id);
CREATE INDEX idx_users_email ON users(email); -- for auth lookup

-- ═══════════════════════════════════════════════════════
-- PROGRESS_RECORDS — Partitioned by recorded_at for
-- high-volume writes. Entity-type polymorphic pattern
-- allows tracking progress on vocab, quizzes, games,
-- lessons without separate tables.
-- ═══════════════════════════════════════════════════════
CREATE TABLE progress_records (
  id           UUID NOT NULL DEFAULT gen_random_uuid(),
  tenant_id    UUID NOT NULL REFERENCES tenants(id),
  user_id      UUID NOT NULL REFERENCES users(id),
  entity_type  VARCHAR(50) NOT NULL, -- 'vocabulary'|'quiz'|'lesson'|'game'
  entity_id    UUID NOT NULL,
  score        SMALLINT,
  metadata     JSONB NOT NULL DEFAULT '{}',
  recorded_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (id, recorded_at)
) PARTITION BY RANGE (recorded_at);

-- Monthly partitions — created by migration or pg_partman
CREATE TABLE progress_records_2024_01 PARTITION OF progress_records
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE INDEX idx_progress_user_tenant ON progress_records(tenant_id, user_id);
CREATE INDEX idx_progress_entity ON progress_records(entity_type, entity_id);

Implementation Phases

19 phases from discovery to enterprise expansion — each with full engineering detail

Goal

Validate the core problem space, define user personas, map domain entities, and identify technical risks before writing a single line of code.

Business Value

Prevents wasted engineering cycles. Aligns the team on vocabulary and boundaries. Produces a shared mental model that survives team growth.

Key Activities

  • Interview 3+ teachers about pain points
  • Map domain using Event Storming
  • Document bounded contexts (Learning, Identity, Content, Analytics)
  • Identify feature flags needed from day 1
  • Define non-functional requirements (latency, availability, data residency)
  • Assess GDPR / COPPA compliance needs

  • Domain glossary document written
  • ERD draft approved

Key Design Decisions

  • UUID everywhere (not integer IDs) — prevents enumeration attacks, works in distributed environments
  • Soft deletes via deleted_at on core entities — audit trail required
  • JSONB for settings and metadata — avoids schema migrations for config changes
  • Polymorphic entity_type / entity_id on progress_records — avoids explosion of progress tables

Security Considerations

  • Never return password_hash from any query
  • All DB connections via connection pool (pg-pool) with connection limits
  • Separate read-only DB user for analytics queries
  • Prisma middleware to auto-inject tenant_id on all queries

SM-2 Algorithm (SRS)

// Simplified SM-2 implementation
function calculateNextReview(card: VocabCard, quality: 0|1|2|3|4|5) {
  if (quality < 3) {
    // Reset — review again soon
    return { interval: 1, ease: card.ease };
  }
  const ease = Math.max(1.3,
    card.ease + 0.1 - (5 - quality) * (0.08 + (5 - quality) * 0.02)
  );
  const interval = card.interval === 0 ? 1
    : card.interval === 1 ? 6
    : Math.round(card.interval * ease);
  return { interval, ease, nextReview: addDays(now(), interval) };
}

Audit Log Design

Every CREATE, UPDATE, DELETE on sensitive tables writes to audit_logs with before/after diff stored as JSONB. Immutable — no UPDATE/DELETE on audit_logs ever.

// Audit log middleware (Prisma)
prisma.$use(async (params, next) => {
  const before = await fetchBefore(params);
  const result = await next(params);
  const after = result;

  if (['create','update','delete'].includes(params.action)) {
    await prisma.auditLog.create({
      data: {
        tenant_id: context.tenantId,
        actor_id: context.userId,
        action: params.action,
        entity_type: params.model,
        entity_id: result.id,
        diff: { before, after },
      }
    });
  }
  return result;
});

// Middleware — check quota before resource creation
async function checkQuota(tenantId: string, resource: string) {
  const tenant = await getTenantFromCache(tenantId);
  const quota = tenant.settings.quotas[resource];
  const current = await countResource(tenantId, resource);

  if (current >= quota) {
    throw new QuotaExceededError({
      resource,
      limit: quota,
      current,
      upgradeUrl: '/billing',
    });
  }
}

Risk Matrix

Identified risks, likelihood, impact, and mitigation strategies

🔴 CRITICAL Data Breach / Tenant Cross-Contamination

Risk: A bug in tenant_id filtering exposes one tenant's data to another.

Likelihood: Medium | Impact: Catastrophic

Mitigation: Prisma middleware auto-injects tenant_id. PostgreSQL RLS as second layer. Integration tests assert cross-tenant isolation. Security audit before each major release.

🟠 HIGH Database Becomes the Bottleneck

Risk: PostgreSQL can't handle 100K concurrent users on a single server.

Likelihood: High (if no action) | Impact: High

Mitigation: Read replica from Phase 12. Aggressive caching. Partition high-volume tables from Phase 1. ClickHouse for analytics offload. Connection pooling with PgBouncer.

🟠 HIGH WebSocket Scaling Under Game Load

Risk: Live game sessions with 50+ students per game overwhelm a single server.

Likelihood: Medium | Impact: High

Mitigation: Redis pub/sub for cross-instance WS message routing. Sticky sessions for WS connections. Load test with k6 before launch. Consider dedicated WS service at scale.

🟡 MEDIUM Vendor Lock-In (S3, Payment)

Risk: Hard dependency on AWS S3 or Stripe makes migration painful.

Likelihood: Low | Impact: Medium

Mitigation: Use S3-compatible interface (MinIO locally, any S3-compat cloud). Abstract storage behind a service class. Stripe abstracted behind PaymentService interface.

🟡 MEDIUM GDPR / COPPA Non-Compliance

Risk: Processing children's data (students under 13) without consent mechanisms.

Likelihood: High if not planned | Impact: Legal/Financial

Mitigation: Teacher is the responsible adult who enrolls students. No direct student marketing. Data export/delete API. Parental consent flow for under-13. DPA with EU customers.

🟢 LOW TypeScript / Prisma Migration Pain

Risk: Prisma schema changes require careful migration management as the team grows.

Likelihood: Medium | Impact: Low (manageable)

Mitigation: Enforce backward-compatible migrations only. Never add NOT NULL without a default. Test all migrations on a clone of production before applying.

Phase Gate Optimization Checklist

Required checks before advancing to the next phase

🛡️ Security Checklist

  • ☐ OWASP Top 10 audit completed
  • ☐ All inputs sanitized and validated (Zod schemas)
  • ☐ SQL injection impossible (Prisma parameterized only)
  • ☐ XSS protection: CSP headers set, React escapes by default
  • ☐ Rate limiting on all auth endpoints
  • ☐ Secrets in environment variables, never in code
  • ☐ HTTPS enforced everywhere
  • ☐ Audit logs capturing all sensitive changes
  • ☐ Tenant isolation test suite passing
  • ☐ Dependency vulnerability scan (npm audit)

⚡ Performance Checklist

  • ☐ All list endpoints paginated (cursor or offset)
  • ☐ N+1 queries eliminated (EXPLAIN ANALYZE verified)
  • ☐ Redis cache hit rate > 70% for hot data
  • ☐ API P95 latency < 200ms under load
  • ☐ Lighthouse score > 90 on main pages
  • ☐ Media files served via CDN (not API)
  • ☐ Database connection pool configured correctly
  • ☐ BullMQ job concurrency tuned
  • ☐ Load test run for expected peak load
  • ☐ Memory leaks checked in long-running processes

🗄️ Database Checklist

  • ☐ All foreign keys indexed
  • ☐ Composite indexes for common query patterns
  • ☐ JSONB columns indexed where queried (GIN)
  • ☐ Partitioned tables have correct partition key
  • ☐ Migration is backward compatible (no breaking changes)
  • ☐ Backup tested and restore verified
  • ☐ Read replica lag < 1 second
  • ☐ Connection pool limits set appropriately
  • ☐ Dead tuples / bloat monitored and autovacuum running

🧪 Testing Checklist

  • ☐ Unit tests for all service functions (> 80% coverage)
  • ☐ Integration tests for all API endpoints
  • ☐ E2E tests for critical user flows (Playwright)
  • ☐ Tenant isolation tests (cross-tenant access = 403)
  • ☐ Load tests for game sessions (50 concurrent players)
  • ☐ Migration tested on production data clone
  • ☐ Error boundary tests for React components
  • ☐ CI runs all tests on every PR

Technology Stack — Decision Records

Every choice with rationale and rejected alternatives

Backend

Fastify
vs Express — 2× faster, built-in schema validation, TypeScript first
TypeScript
Shared types across frontend/backend, prevents runtime errors
Prisma
vs TypeORM — better DX, auto migrations, generated types
BullMQ
vs agenda — Redis-native, better concurrency, observability

Frontend

React + Vite
vs Next.js — simpler, faster dev, SPA sufficient for this use case
React Query
vs SWR — richer API, better caching, mutations + optimistic updates
Zustand
vs Redux — minimal boilerplate, TypeScript native, lightweight
Shadcn/UI
vs MUI — copy-paste ownership, not a dependency, fully customizable

Infrastructure

PostgreSQL
vs MySQL — JSONB, partitioning, RLS, full SQL standard support
Redis
Sorted sets for leaderboards, pub/sub for WS, fast TTL caching
Docker + VPS
vs Kubernetes — cost effective for MVP. Migrate to K8s at 10K+ users
ClickHouse
vs BigQuery — self-hosted, 100× faster aggregations than PostgreSQL
